Why not build an application as a SPA?

The disadvantages of SPAs

The two main disadvantages of SPAs are the initial page load and SEO.

Initial page load

Typically a SPA will load all the things it needs to run the application when the user first lands on the website. Nowadays with SPAs becoming much more established the propriotors have come up with solutions to this problem. In Angular it is very easy to lazy load items so that you are not downloading everything at once, thus drastically reducing the wait time for the user.

Service workers and web workers can be employed to make full use of the browser cache, files can be minimised and tree shaking methods can be used in building the production code reducing the KBs to a speedy response.

SEO

In the early days of the web, Googlebot (Google's search engine crawler) would only crawl the static HTML provided in a page. It was announced in 2014, however, that Googlebot would now attempt to render JavaScript before it began crawling. To help debug any issues with rendering a JavaScript-modified page, Google provided webmasters with the Fetch As Google tool, which shows a snapshot of what Googlebot sees at a particular URL.

There are a number of solutions to optimise a SPAs SEOness, these include pre-rendering and server side rendering. Pre-rendering is where you run the app with a headless browser in your development environment, snapshot the page output, and substitute your HTML files with this snapshot in the server’s response. Server-side rendering (SSR) is where a page is rendered by the web server as part of the server request/response cycle.
